# 문서 거버넌스·기준문서 보고서  
(백엔드 개발자 · DBA · 모듈식 DB 환경)

**목적**: 팀이 커져도 “무엇을 어디까지 문서로 남기고, 누가 갱신하는지”가 흐려지지 않게 **필수 문서 목록**과 **역할**을 정리한다.  
**전제**: DB는 **스키마(모듈) 단위로 경계**가 있고(본 레포는 Prisma `multiSchema`), 백엔드 개발자 전원이 물리·운영·네이밍 철학까지 동일한 수준으로 들고 가기 어렵다는 현실을 인정한다.

**Cursor**: Prisma·마이그레이션 편집 시 에이전트용 짧은 리마인더는 `.cursor/rules/data-platform-alignment.mdc` — 본 문서가 내용의 기준(SSOT)이다.

---

## 1. 왜 API 명세만 떠올리면 부족한가

API 명세(OpenAPI·스웨거·`api-overview.md` 등)는 **계약의 한 축**일 뿐이다. 운영·온보딩·감사·장애 대응에서 필요한 질문은 다음까지 포함된다.

- “이 필드는 **어느 스키마·어느 테이블**에서 오는가?”
- “**누가 쓰기 권한**을 갖는가? (앱 역할 vs DB 역할 vs RLS)”
- “**마이그레이션 순서**와 **다운스트림**은?”
- “**보존 기간·PII**는?”

그래서 **API 명세 + 데이터·운영 문서**가 세트로 있어야 한다.

---

## 2. 백엔드 개발자가 지속 유지·갱신해야 할 문서

| 문서(또는 산출물) | 다루는 내용 | 갱신 시점(최소) |
|-------------------|-------------|-----------------|
| **API 계약** | REST 경로, 메서드, 쿼리/바디, 응답 스키마, `error` 코드, 인증 방식, 버전/브레이킹 정책 | 엔드포인트 추가·변경·폐기 시마다 |
| **API 변경 로그** | 브레이킹 vs 비브레이킹, 클라이언트 영향 | 릴리스 단위 또는 PR 단위 |
| **런북(간단)** | 배포 절차, 필수 env, 롤백, “자주 터지는 401/403” 체크리스트 | 배포 방식이 바뀔 때·장애 후教訓 |
| **아키텍처 스케치** | Next Route Handlers 위치, 미들웨어 matcher, `lib/server` 인증 흐름 | 구조 변경 시 |
| **의존성·통합** | 외부 API(Supabase Auth, 스토리지 등), 웹훅, 큐가 생기면 | 통합 추가·변경 시 |

**형태**: OpenAPI YAML이 이상적이지만, 팀 규모가 작으면 `docs/api-overview.md` + PR에 스키마 스니펫으로 시작해도 된다. 중요한 것은 **“단일 진실 소스가 어디인지”** 한 곳으로 고정하는 것이다.

## 3. DBA(또는 데이터 플랫폼)가 주도·공동 유지하는 문서

DBA가 “테이블만 만진다”가 아니라, **데이터 아키텍처·품질·접근 통제**의 기준을 문서로 남긴다.

| 문서 | 다루는 내용 | 백엔드와의 관계 |
|------|-------------|-----------------|
| **데이터 모델 / ERD** | 엔티티, 관계, 식별자, 주요 제약 | 백엔드는 **읽고** PR에서 위배 시 스스로 질문; DBA/데이터 주도로 **갱신** |
| **스키마·모듈 경계 규칙** | 어떤 도메인이 `core` vs `training` 등 **어느 스키마**에 속하는지, **금지 예** (예: training이 core FK를 역으로 끌어오면 안 됨) | 신규 테이블·마이그레이션 리뷰의 **판정 기준** |
| **네이밍·타입 표준** | 테이블/컬럼/인덱스 명명, `snake_case` vs Prisma `@map`, 금지 타입 | Prisma 스키마 리뷰 시 공통 언어 |
| **마이그레이션 정책** | `deploy` 순서, 롱 트랜잭션, 백필(backfill) 패턴, 다운타임 허용 범위 | 백엔드가 마이그레이션 PR 올릴 때 **체크리스트** |
| **권한·RLS·서비스 롤** | 앱 연결 주체, Supabase RLS 정책 요약, “앱에서 하지 말 것” | API 설계와 **이중 방어** 논의 |
| **데이터 사전(Data dictionary)** | 민감도, PII 여부, 비즈니스 정의, 기본값 의미 | API 필드 설명과 **링크**되면 베스트 |
| **보존·아카이빙** | 로그/이벤트 보존 기간, 삭제 정책 | GDPR/개인정보 이슈 대비 |

모듈식(멀티 스키마)일수록 **“스키마 경계 문서”**가 없으면 백엔드가 임의로 테이블을 넣다가 **순환 의존·권한 혼선**이 나기 쉽다. 이 문서는 **DBA가 소유권을 가지고** 백엔드 리뷰어가 승인 라인에 서는 형태가 안정적이다.

## 4. 공통(제품·엔지니어링)으로 두면 좋은 기준문서

여러 부문이 동시에 참조하는 **한 장짜리**들이다.

| 문서 | 내용 |
|------|------|
| **용어집(Glossary)** | “이벤트”, “세션”, “로스터 상태” 등 제품 용어 = DB/API 용어 매핑 |
| **식별자·멱등 규칙** | UUID 생성 주체, 외부 id, 웹훅 멱등 키 |
| **에러·관측성 규약** | HTTP 코드, `error` 문자열, 로그에 넣을 correlation id |
| **보안 기본선** | 시크릿 저장소, PR에 시크릿 금지, 의존성 취약점 주기 |

## 5. 모듈식 DB일 때 강조할 점 (본 레포 맥락)

Prisma `schemas = ["auth", "core", "development", "management", "training"]`처럼 가면:

1. **스키마 선택 가이드** — “새 기능 X는 기본적으로 스키마 Y”를 한 페이지에 적는다.  
2. **크로스 스키마 FK 정책** — 허용/예외/금지를 명시한다.  
3. **마이그레이션 순서·배포 순서** — “training 테이블이 먼저 없으면 API가 폴백한다” 같은 **운영 사실**을 문서화한다(코드 주석만에 의존하지 않음).  
4. **DBA 리뷰 게이트** — `prisma/migrations` PR에 DBA 또는 데이터 오너 승인을 둘지 여부.

백엔드 개발자의 **이해도가 스키마마다 다를 수 있다**는 전제下에서, **DBA 쪽 문서가 ‘판’**이고 백엔드는 **그 판 위에서 Prisma/API를 짠다**고 역할을 나누는 것이 현실적이다.

## 6. 권장 RACI(간단)

| 활동 | 주 책임(R) | 승인/기준(A) | 협력(C) | 정보(I) |
|------|------------|--------------|---------|---------|
| API 계약·개요 | 백엔드 | EM/테크리드 | 프론트·모바일 | QA |
| 스키마 경계·DDL 표준 | DBA/데이터 | EM | 백엔드 | 보안 |
| 마이그레이션 PR | 백엔드 | DBA(또는 데이터 오너) | — | SRE |
| ERD·사전 | DBA/데이터 | — | 백엔드 | PM |

팀이 작으면 한 사람이 여 역할을 해도 되지만, **문서 종류는 쪼개서** 두는 것이 나중에 분리하기 쉽다.

## 7. 현재 레포 대비 갭(짧게)

| 영역 | 현재 | 권장 |
|------|------|------|

| 에이전트 규칙 | `.cursor/rules/backend-core.mdc` | 유지 + “새 API는 개요 문서 갱신” 명시(이미 권장) |
| README | 현행 스택 반영됨 | API/데이터 문서 링크만 계속 정합 |

## 8. 결론

- **백엔드가 꾸준히 유지할 핵심**: API 계약(및 변경 로그), 배포·인증·미들웨어에 대한 짧은 런북, 아키텍처 스케치.  
- **DBA(데이터)가 꾸준히 유지할 핵심**: ERD·데이터 사전, **모듈(스키마) 경계와 네이밍 규칙**, 마이그레이션·권한·RLS 정책.  
- **공통**: 용어집, 식별자/에러 규약.  
- 모듈식 DB에서는 **“스키마 경계 문서”**가 API 명세만큼 중요하며, 백엔드 이해도 편차를 **문서와 리뷰 게이트**로 메운다고 보면 된다.
