# 코드베이스 성숙도·인상 정리 (초보 틱 vs 전문가 시선)

**목적**: 저장소 구조·파일 배치를 처음 보는 사람(특히 시니어)이 갖기 쉬운 인상을 팀 내부 기준으로 고정해 두기.  
**성격**: 평가·칭찬이 아니라 “지금 상태를 어떻게 읽히는지”에 가깝다.

---

## 1. 초보 단독 과제처럼 보이지 않는 근거

- **Next.js App Router**와 `src/app` 아래 라우트·API(`**/route.ts`) 배치가 일관된 편이다.
- **`src/components/features/<도메인>`**, **`@/lib/config`**, **`src/lib/server`의 인증·팀 스코프 헬퍼**, **Prisma multi-schema** 등은 “페이지 한 파일에만 전부 작성”보다 한 단계 위의 분리다.
- **`docs/`**, **`.cursor/rules`**, 점검·거버넌스 문서가 있으면 **협업·온보딩을 염두에 둔 흔적**으로 읽힌다.
- **미들웨어로 보호 API 게이트**, **빌드·타입이 통과하는 수준**이면 “실제로 돌아가게 만드는 경험”이 있다는 인상을 준다.

→ 전문가가 보면 **“방향은 맞다”**, **레거시에서 끌고 온 제품형 레포**에 가깝다고 읽히기 쉽다.

---

## 2. 성장기·실험 흔적으로 읽힐 수 있는 부분

- **레거시 CSS 트림 + Tailwind 쉘**처럼 **두 축이 공존**하는 구간이 있으면, 첫인상만으로 “최종 스타일 전략이 무엇인지”는 한 번에 안 보일 수 있다.
- **`getTeamDataServices()`(로컬 데모)**와 **DB API(fetch)**가 **같은 제품 안에 병존**하면, 문서를 읽기 전에는 **“의도된 데모 경로인지, 임시인지”**가 헷갈릴 수 있다. (아래 §4 참고)
- 루트에 **`index.html` / `css/` / `js/`** 등 **옛 정적 자산**이 남아 있으면, 저장소만 훑을 때 **메인 진입이 무엇인지** 한 번 헷갈릴 수 있다(README로 보완 가능).
- 문서·규칙이 **최근 정리되기 시작한 티**가 나면 “처음부터 이렇게 설계”라기보다 **운영하면서 다듬는 중**으로 보인다. 이것 자체는 **정상적인 성숙 과정**에 가깝다.

---

## 3. 전문가가 받을 대체적인 느낌 (요약)

| 인상 | 설명 |
|------|------|
| 부정적(초보 단독) | “계획 없이 던져진 한 파일 프로젝트” 느낌은 **보통 나지 않는다**. |
| 중립~긍정 | **실서비스/데모를 목표로 한 Next 전환**, 레거시·DB·API가 겹쳐 **경계와 규칙을 문서로 메우는 단계**로 읽힌다. |
| 자주 따라오는 질문 | API 계약 한 장, 스타일 이행 우선순위, 루트 정적 자산의 **보관 vs 폐기** 정책 등. |

---

## 4. 로컬 `getTeamDataServices` vs DB API — “지금 당장 코드로 통일해야 하나?”

**결론: 지금 당장 전부 API로 몰아넣는 건 ‘애매’할 수 있고, ‘문서로 의도를 박아 두는 것’은 애매하지 않다.**

- **제품적으로**: 팀 코드 없이도 데모·오프라인 느낌을 주려면 **로컬 서비스 경로가 역할이 있다**. 이때 **한 번에 제거하면** 데모 시나리오·온보딩이 줄어든다.
- **엔지니어링적으로**: 혼선의 원인은 “두 경로가 나쁘다”기보다 **“언제 어느 경로인지 한 문장으로 안 적혀 있다”**에 가깝다.

**당장 하기 좋은 것(리스크 낮음)**  
- `README.md` 또는 `docs/data-modes.md` 한 페이지에 예를 들어 다음만 명시한다.  
  - `NEXT_PUBLIC_PLAYPROVE_TEAM_CODE`(또는 `hasPlayproveTeamCode()`)가 **있을 때**: DB API 우선.  
  - **없을 때**: `getTeamDataServices()` 로컬 스토리지 데모.  
  - **같은 화면에서 둘을 섞는 경우**(예: 출결·로스터): **탭/모드 분기**로 의도를 드러낸다.

**나중에 해도 되는 것(범위 큼)**  
- 로컬 경로를 없애고 **항상 API + 시드 DB**만 쓰게 통일 — 환경 구축 비용·시연 방식이 바뀌므로 **제품/운영 합의**가 필요하다.

→ **“지금 해결”의 최선은 코드 대통합이 아니라, README/짧은 doc + (이미 있는) `frontend-core` 규칙 한 줄 보강**으로 의도를 고정하는 쪽이 비용 대비 효과가 크다.
