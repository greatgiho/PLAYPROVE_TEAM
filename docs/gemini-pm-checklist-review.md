# 제미나이 PM식 3대 체크리스트 — 현재 코드베이스 검토 (2026 기준 스냅샷)

외부 조언(데이터 확장성 · 문서화된 로직/타입 · UI 분리)을 **이 레포 실제 소스**에 대입해 정리했다.  
“통과/부분/갭”으로만 적는다.

---

## 1. 데이터 모델의 확장성 (DB / Prisma)

### 멀티 테넌시·N:M

| 항목 | 판정 | 근거 |
|------|------|------|
| 팀 단위 식별 | **통과** | `Team` + `teamCode` 유니크, 하위 `Player.teamId`, `TeamMember.teamId`, `Event.teamId` 등 **팀 FK 패턴**이 일관됨. |
| 유저 ↔ 팀 N:M | **통과** | `TeamMember`에 `teamId` + `userId`, **`@@unique([teamId, userId])`** — 한 유저가 여러 팀 멤버 행을 가질 수 있는 구조. |
| 종목 확장 | **부분** | `Team.sport`가 **문자열**(`default "american_football"`)이라 DB만 보면 **축구·농구 팀 행 추가는 가능**. 다만 앱·Prisma enum(`team_member_role` 등)·화면 카피는 **미식 기준으로 박혀 있을 가능성**이 큼 → “스키마 갈아엎기”보다 **도메인 규칙·UI·API 검증을 종목별로 일반화**할 과제가 남음. |
| JSONB·유연 필드 | **통과** | 다수 모델에 `metadata Json` / `Json?` — 확장 필드용으로 이미 깔려 있음. `Media.metadata` 등도 동일. |

### 갭·권장

- **문서**: “한 유저 다팀·한 팀 다종목”을 **의도로 명시**한 페이지가 없으면(거버넌스 문서 등) 신규 입사자는 앱 코드를 뒤져야 함 → `docs/data-modes.md` 또는 ERD 옆에 **한 단락** 권장.
- **API 레이어**: 라우트가 `teamCode`/세션으로 팀을 고르는 패턴은 있으나, **조직 단위(리그·클럽)** 같은 상위 테넌트는 아직 없음 — 사업이 그쪽으로 가면 **별 설계**.

## 2. 문서화된 비즈니스 로직 (API · 타입)

### Prisma 주석

| 항목 | 판정 | 근거 |
|------|------|------|
| 도메인 필드 설명 | **부분** | `Profile` 일부 필드, `Media`, `EventCoachPlan` 등 **`///` 한국어/영문 주석**이 있음. `Team`/`Player`/`TeamMember` 본문에는 **비즈니스 의도 주석이 거의 없음**. |
| auth 스키마 | **통과(자동)** | Supabase/Prisma가 생성한 RLS·제약 안내 `///` 다수 — **운영 주의**용으로는 좋음. |

**권장**: 조언대로 “기획 의도”를 **`schema.prisma` 상단 + 핵심 모델별 `///`**로 점진 추가(한 PR에 테이블 몇 개씩).

### 타입 안전 (`any`)

| 항목 | 판정 | 근거 |
|------|------|------|
| `: any` / `as any` 남발 | **통과(샘플)** | `src` 전역 grep 기준 **`as any` / `: any` 패턴 미검출**. |
| 경계 타입 | **부분(양호)** | API 파서에서 `unknown` 받고 좁히는 패턴 다수 — **`any`보다 낫다**. |
| 소소한 캐스트 | **갭** | 예: `SessionContext`에서 `as unknown as string` — **한 줄 수준**이나, 나중에 **zod/스키마**로 정리 여지 있음. |

### API 계약 문서

| 항목 | 판정 | 근거 |
|------|------|------|
| 통합 API 개요 | **갭** | `docs/api-overview.md` 또는 OpenAPI **없음** — 인수인계 시 **가장 먼저 물을 곳**. |

## 3. 컴포넌트 분리 (UI vs 로직 · “통파일”)

| 항목 | 판정 | 근거 |
|------|------|------|
| features 폴더 | **통과** | `attendance`, `mypage`, `roster`는 **`use*PageState` + `*PageView`**로 페이지 얇게 분리됨. |
| AppShell | **통과** | 레이아웃 쉘 **전용 컴포넌트**로 분리. |
| 두꺼운 `page.tsx` | **갭** | `dashboard` ~164줄, `login` ~203줄, `injury`, `admin`, `dues` 등 **아직 페이지에 로직·UI 밀집**. |
| 훈련 영역 | **부분** | `CoachPlanPageContent` 등 **대형 단일 컴포넌트** — 담당 분할 시 “차트 담당 / 패널 담당” 나누기엔 **추가 쪼개기** 여지. |

**권장**: Cursor 규칙(`frontend-core`)과 맞춰 **건드리는 화면부터** 훅/뷰 분리; 한 번에 “원자 디자인 전부”는 비용 대비 낮음.

## 총평 (PM 조언과의 정합)

- **1번 DB**: “나중에 갈아엎기” 수준의 구멍은 **당장 보이지 않음**. 다만 **종목·제품 일반화**는 **앱·API 쪽 숙제**로 남아 있음.  
- **2번 문서·타입**: 타입은 **무난**; **Prisma 비즈니스 주석·API 한 장**이 가장 큰 보강 포인트.  
- **3번 UI**: **이미 좋은 방향으로 일부 완료**; 남은 두꺼운 페이지·훈련 덩어리가 **다음 파도**.
